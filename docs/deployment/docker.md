# Docker Deployment

Sentinel ships with two Docker Compose files:

- `docker-compose.yml` — dev infrastructure (PostgreSQL + Redis with TLS)
- `docker-compose.prod.yml` — full production stack (Sentinel + PostgreSQL + Redis, all TLS)

## Dev Compose

The dev compose file starts PostgreSQL and Redis with TLS enabled by default. The FastAPI service runs on the host (via `make start`) for hot-reload.

### Services

| Service | Image | Host Port | Container Port | Transport |
|---------|-------|-----------|----------------|-----------|
| `identity-postgres` | `postgres:16-alpine` | 9001 | 5432 | SSL |
| `identity-redis` | `redis:7-alpine` | 9002 | 6379 | TLS + password auth |

### Key details

- **TLS everywhere**: Both PostgreSQL and Redis use TLS for encrypted connections, even in development. Certificates are auto-generated by `make setup` in `keys/tls/`.
- **Health checks**: Both services have health checks. The Redis health check uses TLS and authenticates with the password.
- **Data persistence**: PostgreSQL data is stored in the `identity_pg_data` Docker volume, so it survives container restarts.
- **Environment**: The service reads its configuration from `service/.env`. See [Environment Variables](environment.md) for the full list.
- **Networking**: All services communicate over the `sentinel-network` bridge network.

### Commands

**Start infrastructure (for local dev):**

```bash
docker compose up -d identity-postgres identity-redis
```

Or just run `make setup`, which starts these automatically.

**View logs:**

```bash
docker compose logs -f identity-postgres
docker compose logs -f identity-redis
```

**Stop everything:**

```bash
docker compose down
```

**Stop and wipe all data:**

```bash
docker compose down -v
```

!!! danger "Data loss"
    The `-v` flag removes the `identity_pg_data` volume, permanently deleting all database contents. Only use this when you want a completely fresh start.

## Production Compose

The `docker-compose.prod.yml` file is designed for production deployment (Docker Compose or Swarm):

- **Docker secrets** for TLS certs and JWT keys (immutable, in-memory)
- **All credentials** injected via `.env.prod` (generated by `make setup`)
- **PostgreSQL SSL**, **Redis TLS + password auth**, plain Redis port disabled
- **`COOKIE_SECURE=true`**, **`DEBUG=false`**, **`BEHIND_PROXY=true`**
- **Resource limits** (CPU + memory) on all containers
- **`restart: unless-stopped`** for crash recovery

### Quick start

```bash
make setup                  # generates keys, certs, .env.prod (one-time)
vim .env.prod               # set BASE_URL, OAuth creds, ADMIN_EMAILS
docker stack deploy -c docker-compose.prod.yml sentinel
```

### Services

| Service | Image | Port | Purpose |
|---------|-------|------|---------|
| `postgres` | `postgres:16-alpine` | internal | Primary datastore (SSL) |
| `redis` | `redis:7-alpine` | internal | Token denylist, rate limiting (TLS + auth) |
| `sentinel` | `ghcr.io/sidxz/sentinel:latest` | 9003 (configurable) | The identity API |

### Docker Secrets

The production compose file mounts sensitive files as Docker secrets (immutable, stored in-memory). These are generated by `make setup`.

| Secret | File Path | Used By |
|--------|-----------|---------|
| `jwt_private_key` | `./keys/private.pem` | Sentinel (JWT signing) |
| `jwt_public_key` | `./keys/public.pem` | Sentinel (JWT verification) |
| `tls_cert` | `./keys/tls/server.crt` | PostgreSQL, Redis (server TLS cert) |
| `tls_key` | `./keys/tls/server.key` | PostgreSQL, Redis (server TLS key) |
| `tls_ca` | `./keys/tls/ca.crt` | Redis, Sentinel (CA certificate for TLS verification) |

Inside the containers, secrets are available at `/run/secrets/<name>`.

See the [Production Checklist](production.md) for the full deployment guide.

## Cross-Service Networking

If you are running other services that need to communicate with Sentinel, they can join the `sentinel-network`:

```yaml
# In another service's docker-compose.yml
services:
  my-service:
    networks:
      - sentinel-network

networks:
  sentinel-network:
    external: true
```

Sentinel will then be reachable at `http://sentinel:9003` from within the Docker network.

## Rebuilding (dev)

After code changes, rebuild the service image:

```bash
docker compose build sentinel
docker compose up -d sentinel
```

Or combine both steps:

```bash
docker compose up -d --build sentinel
```
